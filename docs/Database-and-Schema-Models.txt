# 🗄️ Database Schema & Models

## Project: Campus Women Safety System

---

## 🎯 Overview

This document defines the MongoDB schema design for the system. It is optimized for:

* Real-time operations
* Role-based access
* Case tracking
* Privacy (UID-based identity)

---

## 📁 Collections

```
users
reports
cases
locations
sos_events
```

---

# 👤 1. User Schema

```js
{
  _id: ObjectId,
  uid: String,            // Public identifier (shown in system)
  name: String,           // Hidden from general system
  email: String,
  password: String,
  role: String,           // student | admin | patrol | super_admin
  isActive: Boolean,
  createdAt: Date,
  updatedAt: Date
}
```

### 🔥 Notes

* UID is used everywhere instead of name
* Only admins can access real identity

---

# 📝 2. Report Schema

```js
{
  _id: ObjectId,
  userId: ObjectId,
  uid: String,

  description: String,
  incidentType: String,
  location: {
    lat: Number,
    lng: Number,
    address: String
  },

  evidence: [String],   // file paths

  status: String,       // submitted | under_review | investigating | action_taken | closed

  assignedTo: ObjectId, // admin

  createdAt: Date,
  updatedAt: Date
}
```

---

# 🚨 3. Case Schema

```js
{
  _id: ObjectId,
  reportId: ObjectId,

  type: String,        // report | sos

  status: String,      // active | resolved | closed

  assignedAdmins: [ObjectId],
  assignedPatrols: [ObjectId],

  priority: String,    // low | medium | high | critical

  createdAt: Date,
  updatedAt: Date
}
```

---

# 📍 4. Location Tracking Schema

```js
{
  _id: ObjectId,
  userId: ObjectId,
  caseId: ObjectId,

  coordinates: {
    lat: Number,
    lng: Number
  },

  timestamp: Date
}
```

---

# ⚡ 5. SOS Event Schema

```js
{
  _id: ObjectId,
  userId: ObjectId,
  uid: String,

  caseId: ObjectId,

  status: String,      // active | resolved

  triggeredAt: Date,
  resolvedAt: Date
}
```

---

# 🔗 Relationships

* User → Reports (1:N)
* Report → Case (1:1)
* Case → Patrol/Admin (M:N)
* Case → Locations (1:N)

---

# ⚠️ Design Decisions

## Privacy

* UID used instead of identity
* Sensitive fields restricted

## Performance

* Location stored separately (avoid heavy report docs)

## Real-Time

* Location streamed via sockets, stored periodically

---

# 🚀 Indexing (Important)

```
users.uid (unique)
reports.userId
cases.status
locations.caseId
```
